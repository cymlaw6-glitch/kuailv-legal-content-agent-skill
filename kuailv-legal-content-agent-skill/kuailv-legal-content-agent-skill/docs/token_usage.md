# Token 使用需求说明

「快律」项目需要较高 token plan，原因主要来自以下几个方面。

## 1. 输入文本长

法律材料通常具有高文本长度：

| 输入类型 | 特点 |
|---|---|
| 裁判文书 | 事实、理由、证据、裁判观点较长 |
| 咨询记录 | 多轮对话、事实碎片化 |
| 新闻材料 | 背景信息、当事人表述和舆论较多 |
| 法条原文 | 表达严谨，需要结合上下文解释 |
| 司法解释 | 条款多、适用条件复杂 |

## 2. RAG 上下文消耗大

每次检索通常需要拼接：

- 相关法条；
- 司法解释；
- 类案摘要；
- 裁判规则；
- 平台内容规则；
- 历史模板案例。

这些材料会显著增加输入 token 消耗。

## 3. 多 Agent 连续调用

一次完整任务通常包括：

1. 输入解析；
2. 法律研究；
3. 法律推理；
4. 内容转译；
5. 短视频生成；
6. 合规审查；
7. 质量评分；
8. 安全改写。

每个 Agent 都需要独立上下文和输出，因此会产生多轮 token 消耗。

## 4. 多版本输出

系统需要同时输出：

- 60 秒口播稿；
- 30 秒短版；
- 多个标题；
- 多个封面文案；
- 评论区话术；
- 平台标签；
- 合规风险提示；
- 安全改写版本；
- 质量评分报告。

这会带来较高输出 token 消耗。

## 5. 批量测试需求

项目迭代阶段需要批量测试不同类型案例，例如：

- 婚姻家事；
- 劳动争议；
- 侵权责任；
- 合同纠纷；
- 刑事风险；
- 消费维权；
- 房产继承；
- 网络名誉侵权。

批量测试需要稳定的 token 额度支持。
